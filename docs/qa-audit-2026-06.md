#### `a11y-01` — Search modal has no focus trap; background stays focusable behind the dialog
**Dimension:** Accessibility (WCAG 2.2 AA)  
**Fix path:** `src/assets/js/search.js`

The site-search modal opens as a dialog with role="dialog" aria-modal="true" but search.js never traps focus inside it. open() only sets overlay.hidden=false, adds the body.search-open class, and focuses the input; it never makes the rest of the page inert or aria-hidden, and there is no Tab/Shift+Tab handler. A keyboard user pressing Tab from the close button moves focus into the page content behind the overlay while the modal is still visually on top, and screen-reader users can arrow into the background. This is a WCAG 2.2 AA 2.4.3 (Focus Order) failure for a modal dialog. The only keyboard affordance is Escape to close (search.js line 203). grep confirms no focus-trap, inert, or tabindex management anywhere in search.js.

**Evidence.** src/assets/js/search.js lines 150-170 (open()/close() set only overlay.hidden + body class + input.focus(), no focus containment); src/_includes/search-modal.njk line 7-12 declares role="dialog" aria-modal="true" but base.njk never marks <main> inert when the overlay is open.

**Verification.** Verified against source. The search modal markup (src/_includes/search-modal.njk:8-12) declares role="dialog" aria-modal="true" but search.js never enforces focus containment. open() (src/assets/js/search.js:150-158) only sets overlay.hidden=false, adds body.search-open, lazy-loads pagefind, and focuses the input via requestAnimationFrame; close() (160-170) restores lastFocused. The keydown handler (186-206) handles ONLY Cmd/Ctrl-K, "/", and Escape — there is no Tab/Shift+Tab handler. grep across src/assets/js/ and the layout confirms no inert, no aria-hidden on background, no tabindex management (the only aria-hidden hits are decorative icons). The page wrapper <main id="main"> in src/_layouts/base.njk:192 is never made inert/aria-hidden when the modal opens; the modal is included after <main> at base.njk:203. CSS body.search-open { overflow: hidden; } (site.css:3475) blocks scrolling but not tabbing. Because the dialog sits last in the DOM, after the close button Tab wraps back to the top of the page (skip-link/header/nav/main) which remains fully focusable behind the visible overlay. This is a genuine WCAG 2.4.3 Focus Order (and 4.1.2, given aria-modal asserts containment) failure for a modal dialog. Not a false positive and not handled elsewhere.

#### `a11y-02` — Search results listbox is not keyboard-navigable and the combobox ARIA pattern is incomplete
**Dimension:** Accessibility (WCAG 2.2 AA)  
**Fix path:** `src/assets/js/search.js`

Results render as <ul role="listbox"> with <li role="option"> children (added in renderResults, search.js lines 80-81), and the search input declares aria-controls="search-results" (search-modal.njk line 25). But the listbox/combobox pattern is never completed: the input has no role="combobox", no aria-expanded, and no aria-activedescendant; search.js has no arrow-key handler to move a virtual selection through the options. grep for "activedescendant" and "aria-selected" across src/assets/js and src/_includes returns nothing. The result is a listbox a screen reader announces ("listbox, N options") that cannot be operated with the arrow keys the role implies, and a dangling aria-controls relationship. WCAG 4.1.2 (Name, Role, Value) and the ARIA authoring practices for the combobox/listbox pattern.

**Evidence.** src/assets/js/search.js lines 80-81 (li.setAttribute("role","option")); src/_includes/search-modal.njk line 25 (aria-controls="search-results") + lines 38-42 (role="listbox"); no aria-activedescendant/aria-selected/arrow-key code anywhere.

**Verification.** Verified all cited evidence holds. search.js:81 sets role="option" on each <li>; search-modal.njk:38-42 declares the <ul id="search-results" role="listbox">; search-modal.njk:25 sets aria-controls="search-results" on the search input. The negatives are confirmed too: rg across src/assets/js and src/_includes finds zero aria-activedescendant and zero aria-selected; the input has no role="combobox" and no aria-expanded (the only aria-expanded hits are in nav.njk/theme.js/person-card.njk, unrelated to search); and search.js has no ArrowUp/ArrowDown handler at all — its sole keydown listener (lines 186-206) handles only Cmd/Ctrl-K, "/", and Escape. So the listbox/combobox ARIA pattern is genuinely incomplete: a screen reader announces "listbox, N options" that cannot be driven with the arrow keys the role implies, and aria-controls points at a listbox whose controller lacks combobox role and state. This is a legitimate WCAG 4.1.2 (Name, Role, Value) defect. Severity tempered to medium rather than higher because the result items are real <a class="search-result-link"> anchors (search.js:83-85, 124-125), so keyboard users are not blocked — they can Tab to and activate the links normally; the harm is incorrect/incomplete AT semantics rather than a loss of operability or a keyboard trap.

#### `a11y-03` — Search option elements wrap a focusable link, an invalid listbox-option composition
**Dimension:** Accessibility (WCAG 2.2 AA)  
**Fix path:** `src/assets/js/search.js`

Each rendered result is <li role="option"> containing an interactive <a class="search-result-link"> (search.js lines 81-125). ARIA forbids interactive/focusable descendants inside a role="option"; an option is meant to be a non-interactive selectable leaf. Here Tab lands on the nested <a> (because it is the real focusable element) while the wrapping option is what the listbox role exposes, so the announced role and the actually-operable element diverge. Either drop the listbox/option roles and treat the results as a plain navigable list of links, or move the href/selection onto the option and remove the nested anchor. WCAG 1.3.1 / 4.1.2.

**Evidence.** src/assets/js/search.js lines 80-126: li.setAttribute("role","option") then a=document.createElement("a"); a.href=d.url; ... li.appendChild(a).

**Verification.** Verified against source. src/_includes/search-modal.njk:38-42 declares the results container as <ul role="listbox">, and src/assets/js/search.js:79-126 builds each result as <li role="option"> wrapping an interactive <a class="search-result-link" href>. ARIA forbids focusable/interactive descendants inside role="option" (an option must be a non-interactive selectable leaf), so the announced role (listbox/option) diverges from the actually-operable element (the nested anchor reached by Tab). Crucially, no real listbox keyboard model exists: search.js has no aria-activedescendant, no aria-selected, no ArrowUp/ArrowDown handling, and the input is not a role="combobox" (it only has aria-controls). The listbox/option roles are therefore vestigial and misrepresent the widget. This is a genuine WCAG 1.3.1 / 4.1.2 issue, not a false positive, and not handled elsewhere. Fix: either drop the listbox/option roles and treat results as a plain list of links, or implement a true combobox/listbox keyboard pattern with selection on the option itself. Severity medium is appropriate: results remain operable via Tab and links are correctly named, so it is a misannouncement/comprehension hazard for AT users rather than a total block.

#### `i18n-02` — Board bios render English form-submitted text on FR/DE pages with no lang="en" wrapper
**Dimension:** i18n faithfulness (EN vs FR/DE)  
**Fix path:** `src/_includes/person-card.njk`

src/_includes/person-card.njk renders person.bio, person.themes, and person.affiliation as raw HTML (| safe) on all three locales with no lang annotation. On _site/board.fr.html and _site/board.de.html the bios are full English paragraphs inside French/German chrome, e.g. "John Helferich is a Lecturer in Politics at Hertford College and a researcher with the Oxford Global Security Programme." and "Eugenio assists with the development of our communication strategy and the logistics of our events.". The research-themes label is localised (t.boardPage.researchThemes renders "Forschungsthemen:" on DE) but the value stays English: _site/board.de.html shows "Forschungsthemen: Domestic determinants of security policy, Role of ideas, norms, and institutions...". Because these bios are author-supplied via the submission Form (board.json), hand-translation is not expected per house rule, but the absence of lang="en" on foreign-language runs is a real i18n/accessibility correctness issue: screen readers on the FR/DE pages mispronounce the English bios. Same root cause as i18n-01 (synced/submitted data emitted without language annotation).

**Evidence.** src/_includes/person-card.njk:102 `<p class="person-bio">{{ person.bio | safe }}</p>` (no lang), :117 `<strong>{{ t.boardPage.researchThemes }}:</strong> {{ person.themes | safe }}` (label localised, value not), :97 person.affiliation likewise. Built evidence: _site/board.fr.html and _site/board.de.html contain "John Helferich is a Lecturer in Politics..." and _site/board.de.html "Forschungsthemen: Domestic determinants of security policy...".

**Verification.** Verified against source and built HTML. src/_includes/person-card.njk renders person.bio (:102), person.themes value (:117, label localised via t.boardPage.researchThemes but value passed raw), person.affiliation (:97), and the actually-used person.position (:88) / person.institution (:91) all via `| safe` with no lang annotation. Built _site/board.fr.html contains the full English Helferich bio ("John Helferich is a Lecturer in Politics at Hertford College...") and English positions ("CNRS Research Fellow", "Assistant Professor of War Studies") inside a page whose root is <html lang="fr">; _site/board.de.html under <html lang="de"> renders "Forschungsthemen:</strong> International Relations, Foreign Policy Analysis..." (localised label, English value). I confirmed the only 3 lang="en" occurrences on board.de.html are hreflang link attributes, not content lang annotations, so there is zero lang wrapping on these English runs. This is a genuine WCAG 3.1.2 (Language of Parts, AA) failure: screen readers on the FR/DE pages will pronounce the English bios/positions/themes using the page locale. The house rule exempting hand-translation of Form-submitted bios does not address the missing lang attribute, so it does not neutralise the finding. One inaccuracy in the candidate: the affiliation path (:97) is not exercised because the board data uses the position/institution shape (:88/:91); but those lines share the identical defect and do render English, so the core claim holds.

#### `seo-01` — terms.html and refund.html are byte-identical indexable duplicate pages

**Verification.** Verified. src/refund.njk (line 10) and src/terms.njk (line 15) both `{% include "terms-body.njk" %}` with identical front-matter title ("Terms & Conditions") and identical description. Built _site/terms.html and _site/refund.html have byte-identical <main> bodies (2095 chars each, compared equal — note the finding's "1236 chars" figure is wrong, but the "byte-identical" claim is correct), identical <title>Terms &amp; Conditions — EISS</title>, identical meta description, no noindex/meta-robots on either, and each carries its own self-canonical (terms.html→/terms.html, refund.html→/refund.html) so they do not consolidate. Added context: refund.html is the live page the site footer actually links to (footer.njk line 86 → /refund.html), and the visual sitemap documents it as a deliberate alternate ("also at /refund.html"), so it exists for legacy-URL preservation — intentional in origin but not addressed for SEO (no 301, no cross-canonical, no noindex). #### `seo-02` — index.html canonical (/) does not match its own en / x-default hreflang (/index.html)

**Fix path:** `src/index.njk`

On the English homepage the canonical and og:url resolve to https://eiss-europa.com/ (bare root, matching the sitemap.xml <loc>https://eiss-europa.com/</loc>), but the hreflang self-reference and x-default both point to https://eiss-europa.com/index.html. The FR and DE index pages likewise list the en alternate as /index.html, so the entire cluster references a URL the homepage does not consider canonical. This is the only canonical-vs-self-hreflang mismatch on the site (all other pages use .html consistently for both).

**Evidence.** _site/index.html: <link rel="canonical" href="https://eiss-europa.com/"> and <meta property="og:url" content="https://eiss-europa.com/"> but <link rel="alternate" hreflang="en" href="https://eiss-europa.com/index.html"> and hreflang="x-default" href="https://eiss-europa.com/index.html". sitemap.xml line 49: <loc>https://eiss-europa.com/</loc>. Source: src/index.njk lines 9-12 (alternates.en: /index.html); src/_layouts/base.njk lines 44-48.

**Verification.** Verified against the built HTML and source. _site/index.html line 13 canonical = "https://eiss-europa.com/" and line 34 og:url = "https://eiss-europa.com/" (bare root), but line 17 emits hreflang en and x-default = "https://eiss-europa.com/index.html". The sitemap (_site/sitemap.xml line 49) lists the homepage as <loc>https://eiss-europa.com/</loc>, matching the canonical, not the hreflang. Root cause confirmed: src/_layouts/base.njk line 34 builds canonical from {{ site.url }}{{ page.url }} (page.url for permalink /index.html renders as /), while lines 46 and 48 build the hreflang/x-default from alternates.en, which src/index.njk line 10 hardcodes as /index.html. A site-wide scan of every EN page confirmed index.html is the only page where the canonical differs from its own en hreflang; all other pages (permalink /foo.html → page.url /foo.html) are consistent.
